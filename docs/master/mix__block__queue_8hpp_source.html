36 /*!
37  * Implements reading an unordered sequence of items from multiple workers,
38  * which sends Blocks. This class is mainly used to implement MixChannel.
39  *
40  * When Blocks arrive from the net, the Multiplexer pushes (src, Blocks) pairs
41  * to MixChannel, which pushes them into a MixBlockQueue. The
42  * MixBlockQueue stores these in a ConcurrentBoundedQueue for atomic reading.
43  *
44  * When the MixChannel should be read, MixBlockQueueReader is used, which
45  * retrieves Blocks from the queue. The Reader contains one complete BlockReader
46  * for each inbound worker, and these BlockReaders are attached to BlockQueue
47  * instances inside the MixBlockQueue.
48  *
49  * To enable unordered reading from multiple workers, the only remaining thing
50  * to do is to fetch Blocks from the main mix queue and put them into the
51  * right BlockQueue for the sub-readers to consume. By taking the Blocks from
52  * the main mix queue, the Reader only blocks when no inbound Blocks are
53  * available.
54  *
55  * To enable switching between items from different workers, the
56  * MixBlockQueueReader keeps track of how many _whole_ items are available on
57  * each reader. This number is simply -1 of the number of items known to start
58  * in the received blocks. The last item _may_ span further Blocks, and cannot
59  * be fetched without infinitely blocking the sub-reader, since no thread will
60  * deliver the next Block.
61  */

188 /*!
189  * Reader to retrieve items in unordered sequence from a MixBlockQueue. This
190  * is not a full implementation of _all_ methods available in a normal
191  * BlockReader. Mainly, this is because only retrieval of _whole_ items are
192  * possible. Due to the unordered sequence, these probably have to be all of
193  * equal type as well.
194  *
195  * The Reader supports all combinations of consuming and keeping. However, do
196  * not assume that the second round of reading delivers items in the same order
197  * as the first. This is because once items are cached inside the BlockQueues of
198  * MixBlockQueue, we use a plain CatReader to deliver them again (which is
199  * probably faster as it has a sequential access pattern).
200  *
201  * See \ref MixBlockQueue for more information on how items are read.
202  */
203 class MixBlockQueueReader
204 {
205  static constexpr bool debug = false;
206 
207 public:
208  using CatBlockSource = data::CatBlockSource<DynBlockSource>;
209  using CatBlockReader = BlockReader<CatBlockSource>;

212  bool consume, size_t local_worker_id);

214  //! non-copyable: delete copy-constructor

216  //! non-copyable: delete assignment operator
217  MixBlockQueueReader& operator = (const MixBlockQueueReader&) = delete;
218  //! move-constructor: default
219  MixBlockQueueReader(MixBlockQueueReader&&) = default;
220  //! move-assignment operator: default
221  MixBlockQueueReader& operator = (MixBlockQueueReader&&) = default;
222 
223  //! Possibly consume unread blocks.
224  ~MixBlockQueueReader();
225 
226  //! HasNext() returns true if at least one more item is available.
227  bool HasNext() {
228  if (reread_) return cat_reader_.HasNext();
229 
230  if (available_) return true;
231  if (open_ == 0) return false;
232 
233  return PullBlock();
234  }
235 
236  //! Next() reads a complete item T
237  template <typename T>
238  T Next() {
239  assert(HasNext());
240 
241  if (reread_) {
242  return cat_reader_.template Next<T>();
243  }
244  else {
245  if (!available_) {
246  if (!PullBlock()) {
247  throw std::runtime_error(
248  "Data underflow in MixBlockQueueReader.");
249  }
250  }
251 
252  assert(available_ > 0);
253  assert(selected_ < readers_.size());
254 
255  --available_;
256  return readers_[selected_].template Next<T>();
257  }
258  }
