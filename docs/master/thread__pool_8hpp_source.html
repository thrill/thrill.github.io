30 /*!
31  * ThreadPool starts a fixed number p of std::threads which process Jobs that
32  * are \ref Enqueue "enqueued" into a concurrent job queue. The jobs
33  * themselves can enqueue more jobs that will be processed when a thread is
34  * ready.
35  *
36  * The ThreadPool can either run until
37  *
38  * 1. all jobs are done AND all threads are idle, when called with
39  * LoopUntilEmpty(), or
40  *
41  * 2. until Terminate() is called when run with LoopUntilTerminate().
42  *
43  * Jobs are plain std::function<void()> objects (actually: common::delegate),
44  * hence the pool user must pass in ALL CONTEXT himself. The best method to pass
45  * parameters to Jobs is to use lambda captures. Alternatively, old-school
46  * objects implementing operator(), or std::binds can be used.
47  *
48  * The ThreadPool uses a condition variable to wait for new jobs and does not
49  * remain busy waiting.
50  *
51  * Note that the threads in the pool start **before** the two loop functions are
52  * called. In case of LoopUntilEmpty() the threads continue to be idle
53  * afterwards, and can be reused, until the ThreadPool is destroyed.
54 
55 \code
56 ThreadPool pool(4); // pool with 4 threads
57 
58 int value = 0;
59 pool.Enqueue([&value]() {
60  // increment value in another thread.
61  ++value;
62 });
63 
64 pool.LoopUntilEmpty();
65 \endcode
66 
67  * ## Synchronization Primitives
68  *
69  * Beyond threads from the ThreadPool, the framework contains two fast
70  * synchronized queue containers:
71  *
72  * - \ref ConcurrentQueue
73  * - \ref ConcurrentBoundedQueue.
74  *
75  * If the Intel Thread Building Blocks are available, then these use their
76  * lock-free implementations, which are very fast, but do busy-waiting for
77  * items. Otherwise, compatible replacements are used.
78  *
79  * The \ref ConcurrentQueue has no busy-waiting pop(), only a try_pop()
80  * method. This should be preferred! The \ref ConcurrentBoundedQueue<T> has a
81  * blocking pop(), but it probably does busy-waiting.
82  */
